# 新增指标计算修复报告

## 📋 问题背景

用户询问：
> "这个计算的对吗？"（指向新增指标计算代码）

**检查发现的问题**：
1. **强势度计算**：`涨幅 / 振幅` 在下跌时会产生负数，不合理
2. **价格位置计算**：✅ 正确，无需修改
3. **涨速计算**：使用 `涨幅 / 4`，但涨幅是相对于昨日收盘价，不准确

## 🔍 问题分析

### 问题1：强势度计算不合理

**原始代码**：
```python
# 1. 强势度 = 涨幅 / 振幅（避免冲高回落）
strength_ratio = (pc / zf) if zf > 0 else 0
```

**问题**：
- `pc` = 涨跌幅（%），可能是负数
- `zf` = 振幅（%），总是正数
- **当股票下跌时，强势度是负数**，不符合"强势度"的语义

**示例**：
```
股票A：涨幅8%，振幅10% → 强势度 = 0.8（冲高回落）
股票B：涨幅8%，振幅8% → 强势度 = 1.0（完美强势）
股票C：涨幅-5%，振幅10% → 强势度 = -0.5 ❌（负数不合理）
```

### 问题2：涨速计算不准确

**原始代码**：
```python
# 3. 涨速 = 涨幅 / 4小时（判断拉升速度）
price_velocity = pc / 4.0
```

**问题**：
- `pc` = 涨跌幅（相对于**昨日收盘价**）
- **不是相对于今日开盘价**
- 例如：昨日收盘10元，今日开盘11元（集合竞价涨10%），当前11元
  - 涨幅 = 10%
  - 涨速 = 10% / 4 = 2.5%/小时 ❌
  - **实际上开盘后涨幅是0%，涨速应该是0**

## ✅ 解决方案

### 方案1：修复强势度计算

**新逻辑**：
```python
# 强势度 = 价格位置（当前价在当日振幅中的位置）
# 范围：0-1，越接近1越强势（收盘价接近最高价）
strength_ratio = ((p - l) / (h - l)) if (h - l) > 0 else 0.5
```

**优点**：
- ✅ 总是正数（0-1之间）
- ✅ 语义清晰：0.9表示强势（接近最高价），0.1表示弱势（接近最低价）
- ✅ 适用于上涨和下跌股票

**示例**：
```
股票A：最低10元，最高20元，当前19元 → 强势度 = 0.9（强势）
股票B：最低10元，最高20元，当前11元 → 强势度 = 0.1（弱势）
股票C：最低10元，最高20元，当前15元 → 强势度 = 0.5（中性）
```

### 方案2：修复涨速计算

**新逻辑**：
```python
# 涨速 = 开盘后涨幅 / 交易时长（判断拉升速度）
# 使用开盘价计算，更准确反映盘中拉升速度
if o > 0:
    intraday_change = ((p - o) / o) * 100  # 开盘后涨幅（%）
    price_velocity = intraday_change / 4.0  # 每小时涨速（%/小时）
else:
    # 如果没有开盘价，降级使用涨幅/4
    price_velocity = pc / 4.0
```

**优点**：
- ✅ 使用开盘价，准确反映盘中拉升速度
- ✅ 券商数据有开盘价字段 `o`，无需额外请求K线
- ✅ 有降级方案，兼容没有开盘价的情况

**示例**：
```
昨日收盘10元，今日开盘11元，当前12元
- 涨幅（相对昨收）= 20%
- 开盘后涨幅 = (12-11)/11 = 9.09%
- 涨速 = 9.09% / 4 = 2.27%/小时 ✅（准确）

原始计算：
- 涨速 = 20% / 4 = 5%/小时 ❌（不准确）
```

## 🎯 为什么不用K线数据？

用户提问：
> "为什么不能去拿K线呢，是不是太麻烦了？毕竟这么多股票"

**回答**：
1. **性能问题**：筛选时可能有几千只股票，逐个请求K线数据会很慢
2. **API限制**：频繁请求可能触发限流
3. **实时性**：K线数据可能有延迟

**解决方案**：
- ✅ **券商实时数据里已经有开盘价字段 `o`**
- ✅ 无需额外请求K线，直接使用即可
- ✅ 实时性好，性能高

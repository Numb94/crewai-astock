# 社区情绪全局锁优化报告

**优化时间**: 2025-12-10  
**优化目标**: 避免批量分析时并发调用社区情绪API导致账号被封  
**优化方案**: 在 `get_stock_community_comments` 工具中添加全局锁

---

## 📋 问题背景

### 原有问题
- **批量分析并发**: 使用 `analyze_stocks_parallel` 分析5只股票时，会同时提交5个 `get_stock_community_comments` 任务
- **ThreadPoolExecutor并发**: 这5个任务会同时执行，导致5个并发请求到MCP服务器
- **账号被封**: 短时间内大量并发请求触发反爬虫机制，导致账号被封

### 用户需求
- 批量分析多只股票时，社区情绪API应该串行调用
- 其他维度（技术面、资金面等）可以继续并行
- 降低API调用频率，避免账号被封

---

## ✅ 优化方案

### 核心思路：全局锁 + 双重检查锁定

**实现位置**: `src/agents/tools/community_sentiment_tools.py`

**核心代码**:
```python
# 全局锁：确保同一时间只有一个社区情绪请求在执行
_community_request_lock = threading.Lock()

@tool("获取股票社区评论")
def get_stock_community_comments(stock_code: str, max_items: int = 5) -> str:
    # 1. 检查缓存（无锁）
    if cache_key in _community_comments_cache:
        return cached_data['result']
    
    # 2. 获取锁（串行执行）
    with _community_request_lock:
        # 3. 双重检查缓存（避免重复请求）
        if cache_key in _community_comments_cache:
            return cached_data['result']
        
        # 4. 调用MCP API
        comments = asyncio.run(client.get_all_platforms_comments(...))
        
        # 5. 延迟后释放锁（避免下一个请求立即执行）
        time.sleep(_request_delay)
```

**优点**:
- ✅ 简单高效：只需添加一个全局锁
- ✅ 不影响其他维度：技术面、资金面等仍然并行
- ✅ 双重检查：避免等待锁期间的重复请求
- ✅ 延迟释放：确保请求间隔，进一步降低风险

## 📊 工作原理

### 场景1: 批量分析5只股票（首次调用）

```
ThreadPoolExecutor 提交5个任务:
├─ 股票1: 技术面 || 资金面 || 基本面 || 新闻 || 逐笔 || [社区情绪-等待锁]
├─ 股票2: 技术面 || 资金面 || 基本面 || 新闻 || 逐笔 || [社区情绪-等待锁]
├─ 股票3: 技术面 || 资金面 || 基本面 || 新闻 || 逐笔 || [社区情绪-等待锁]
├─ 股票4: 技术面 || 资金面 || 基本面 || 新闻 || 逐笔 || [社区情绪-等待锁]
└─ 股票5: 技术面 || 资金面 || 基本面 || 新闻 || 逐笔 || [社区情绪-等待锁]

社区情绪串行执行（全局锁保护）:
1. 股票1获取锁 → 调用API → 延迟1.5秒 → 释放锁
2. 股票2获取锁 → 调用API → 延迟1.5秒 → 释放锁
3. 股票3获取锁 → 调用API → 延迟1.5秒 → 释放锁
4. 股票4获取锁 → 调用API → 延迟1.5秒 → 释放锁
5. 股票5获取锁 → 调用API → 延迟1.5秒 → 释放锁

总耗时: 5 × (API调用2秒 + 延迟1.5秒) = 17.5秒
```

### 场景2: 批量分析5只股票（缓存命中）

```
社区情绪直接返回缓存（无需锁）:
1. 股票1 → 缓存命中 → <0.1秒
2. 股票2 → 缓存命中 → <0.1秒
3. 股票3 → 缓存命中 → <0.1秒
4. 股票4 → 缓存命中 → <0.1秒
5. 股票5 → 缓存命中 → <0.1秒

总耗时: <0.5秒
```

## 🎯 配置说明

### COMMUNITY_CACHE_TTL（缓存时间）

**默认值**: `3600`（1小时）

**推荐配置**:
- `3600` (1小时) - 默认，平衡性能和实时性
- `7200` (2小时) - 更安全，减少API调用
- `1800` (30分钟) - 更实时，但API调用更频繁

---

### COMMUNITY_REQUEST_DELAY（请求间隔）

**默认值**: `1.5`（1.5秒）

**推荐配置**:
- `1.5` (1.5秒) - 默认，平衡速度和安全
- `2.0` (2秒) - 更安全，但速度更慢
- `1.0` (1秒) - 更快，但风险稍高

**作用**: 在释放锁之前延迟，确保下一个请求不会立即执行
